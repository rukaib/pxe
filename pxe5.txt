# Real Reasons Why PXE Server WON'T Work (Brutal Honesty)

## ❌ **Top Reasons for FAILURE**

### 1. **UEFI Secure Boot - The #1 Killer** 🔒

```
Success Rate with Secure Boot ON:   5% ❌
Success Rate with Secure Boot OFF: 95% ✅

THE PROBLEM:
Modern PCs (2012+) have UEFI Secure Boot enabled by default.
Secure Boot ONLY allows signed bootloaders.
Your custom PXE boot files are NOT signed by Microsoft.

RESULT:
┌─────────────────────────────────────┐
│ Attempting to boot from network...  │
│ Verifying boot file signature...    │
│ ⚠️  Secure Boot Violation           │
│ Boot failed.                         │
└─────────────────────────────────────┘
```

**Why This Fails:**
```
Your bootx64.efi → Unsigned
Client firmware → Rejects unsigned files
No boot → Nothing happens

Microsoft-signed shim → Requires $$$$ certificate
Community workarounds → Complex, unreliable
```

**The Fix:**
```
YOU MUST disable Secure Boot on EVERY client PC
- Enter BIOS/UEFI (F2, Del, F12 during boot)
- Find "Secure Boot" setting
- Disable it
- Save and exit

If you can't disable Secure Boot (corporate lockdown):
→ PXE boot WILL NOT WORK ❌
```

### 2. **Network Drivers Missing in WinPE** 🌐

```
Boot Success Rate by Network Adapter:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Intel Ethernet:        90% ✅
Realtek Ethernet:      70% ⚠️
Broadcom Ethernet:     60% ⚠️
Killer Ethernet:       40% ❌
USB-to-Ethernet:       20% ❌
WiFi adapters:          0% ❌
```

**THE PROBLEM:**
```
PXE Boot works → WinPE loads → Looks good!
                                    ↓
                          startnet.cmd runs
                                    ↓
                          "net use Z: \\server\share"
                                    ↓
                          ❌ ERROR: Network path not found
```

**Why:**
```
WinPE (boot.wim) includes BASIC drivers only
Your specific network adapter driver → NOT included
No network driver → Can't connect to SMB share
Can't access share → Can't run setup.exe
Installation FAILS
```

**What You'll See:**
```cmd
X:\Windows\System32> net use Z: \\192.168.1.1\WinInstall
System error 53 has occurred.
The network path was not found.

X:\Windows\System32> ipconfig
Ethernet adapter Ethernet0:
   Media State . . . . . . . . . : Media disconnected

← Network adapter not recognized!
```

**The Fix Requires Windows Machine:**
```powershell
# 1. Download driver for your specific NIC
# Example: Intel I225-V driver from intel.com

# 2. Mount boot.wim
Dism /Mount-Wim /WimFile:C:\WinPE\sources\boot.wim /index:1 /MountDir:C:\mount

# 3. Inject driver
Dism /Image:C:\mount /Add-Driver /Driver:C:\Drivers\Intel_I225V /Recurse

# 4. Save and unmount
Dism /Unmount-Wim /MountDir:C:\mount /Commit

# 5. Copy updated boot.wim to PXE server
```

**Problem:**
```
You need to do this for EVERY different network card model!

Office with 10 different PC models = 10 different driver sets
New PC arrives = Re-inject drivers
Maintenance nightmare
```

### 3. **DHCP Conflicts** 🔄

```
Scenario: You already have a router on the network

Your Network:
┌─────────────┐      ┌──────────────┐      ┌─────────┐
│ Your Router │──────│ Your PXE     │──────│ Client  │
│ DHCP: ON    │      │ Server       │      │ PC      │
│ 192.168.1.1 │      │ DHCP: ON     │      │         │
└─────────────┘      │ 192.168.1.2  │      └─────────┘
                     └──────────────┘

What happens:
1. Client: "I need an IP address!"
2. Router: "Here's 192.168.1.50, boot from me!"
3. PXE Server: "No! Here's 192.168.1.100, boot from me!"
4. Client: "??? Confused, picking random one..."
5. Result: Boots from wrong DHCP → No PXE boot file → FAIL
```

**Real-World Example:**
```
I've seen this fail 100 times:

User sets up PXE server
Doesn't disable router DHCP
Client gets IP from router (no boot file)
PXE never even attempts
User thinks "PXE doesn't work"

It DOES work, just never got triggered!
```

**Solutions:**

**Option A: Disable router DHCP (Best)**
```
1. Log into your router (192.168.1.1)
2. Find DHCP settings
3. Disable DHCP server
4. Let PXE server handle everything
```

**Option B: Use DHCP Option 66/67 on router**
```
Some routers support PXE options:
- Option 66: TFTP server IP (192.168.1.2)
- Option 67: Boot filename (pxeboot.0)

But many consumer routers DON'T support this ❌
```

**Option C: Separate network (Isolation)**
```
Direct cable: Server ←→ Client
No router involved
Guaranteed to work ✅
But limits to 1 client at a time
```

### 4. **Firewall Blocking** 🔥

```
Windows Firewall (on PXE server) blocks:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Port 67 UDP  → DHCP responses   ❌
Port 69 UDP  → TFTP transfers   ❌
Port 445 TCP → SMB file sharing ❌
Port 139 TCP → NetBIOS          ❌

Linux firewall (ufw/iptables):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Same ports blocked by default    ❌
```

**What You See:**
```
Client PC:
"PXE-E32: TFTP open timeout"
or
"PXE-E3B: TFTP Error - File not found"

Server logs show:
"TFTP request from 192.168.1.100"
But client never receives response

← Firewall is silently dropping packets
```

**The Fix (Linux):**
```bash
# Disable firewall completely (testing)
sudo ufw disable

# OR allow specific ports
sudo ufw allow 67/udp   # DHCP
sudo ufw allow 68/udp   # DHCP
sudo ufw allow 69/udp   # TFTP
sudo ufw allow 80/tcp   # HTTP
sudo ufw allow 445/tcp  # SMB
sudo ufw allow 139/tcp  # NetBIOS
```

**The Fix (Windows):**
```powershell
# Disable Windows Firewall (testing)
Set-NetFirewallProfile -Profile Domain,Public,Private -Enabled False

# OR create rules
New-NetFirewallRule -DisplayName "DHCP Server" -Direction Inbound -Protocol UDP -LocalPort 67 -Action Allow
New-NetFirewallRule -DisplayName "TFTP Server" -Direction Inbound -Protocol UDP -LocalPort 69 -Action Allow
New-NetFirewallRule -DisplayName "SMB" -Direction Inbound -Protocol TCP -LocalPort 445 -Action Allow
```

### 5. **BCD (Boot Configuration Data) Corruption** 💾

```
THE NIGHTMARE SCENARIO:

WinPE boots successfully ✅
Shows "Starting Windows..." ✅
Then... BLACK SCREEN forever ❌

Or:
"BOOTMGR is missing"
"winload.exe is missing"
"0xc000000f: Boot selection failed"
```

**Why This Happens:**
```
BCD file has HARDCODED paths:
\Device\RamDisk{UUID}\sources\boot.wim
\Boot\BCD
\Boot\boot.sdi

If paths don't EXACTLY match your setup:
→ Boot fails
→ No error message
→ Just hangs or crashes
```

**The Problem:**
```
BCD is BINARY format
Can't just edit with notepad
Needs bcdedit.exe (Windows only)
Even small typo = total failure
GUIDs must match exactly
```

**Example of Required BCD Fix:**
```powershell
# This is what you MUST do (on Windows):
bcdedit /store C:\tftp\Boot\BCD /set {default} ramdisksdidevice boot
bcdedit /store C:\tftp\Boot\BCD /set {default} ramdisksdipath \Boot\boot.sdi
bcdedit /store C:\tftp\Boot\BCD /set {default} osdevice ramdisk=[boot]\sources\boot.wim,{ramdiskoptions}
bcdedit /store C:\tftp\Boot\BCD /set {default} device ramdisk=[boot]\sources\boot.wim,{ramdiskoptions}
bcdedit /store C:\tftp\Boot\BCD /set {default} systemroot \windows
bcdedit /store C:\tftp\Boot\BCD /set {default} detecthal yes
bcdedit /store C:\tftp\Boot\BCD /set {default} winpe yes

# Get ONE character wrong? Doesn't boot. No explanation.
```

**Many people give up here** because debugging BCD is nearly impossible without deep Windows knowledge.

### 6. **BIOS vs UEFI Boot Mode Mismatch** ⚙️

```
Your Setup:        Client PC:        Result:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BIOS boot files   UEFI mode         FAIL ❌
UEFI boot files   BIOS mode         FAIL ❌
BIOS boot files   BIOS mode         WORKS ✅
UEFI boot files   UEFI mode         WORKS ✅
```

**THE PROBLEM:**
```
You create BIOS PXE boot (pxelinux.0)
Client PC is in UEFI mode
Client requests "bootx64.efi"
Server only has "pxelinux.0"
→ Boot file not found → FAIL
```

**What Makes This Confusing:**
```
Same PC can boot in EITHER mode!
UEFI mode: Uses .efi files
BIOS mode: Uses .0 or .n12 files

User thinks: "Why does it work on this PC but not that one?"
Answer: Different boot modes!
```

**The Fix:**
```bash
# You need BOTH sets of files:

/srv/tftp/
├── pxelinux.0              ← BIOS boot
├── ldlinux.c32             ← BIOS boot
├── Boot/
│   ├── bootx64.efi         ← UEFI boot
│   ├── bootmgfw.efi        ← UEFI boot
│   └── BCD                 ← UEFI boot config

# AND dnsmasq must detect and serve correct file:
dhcp-match=set:bios,option:client-arch,0
dhcp-match=set:efi64,option:client-arch,7
dhcp-boot=tag:bios,pxelinux.0
dhcp-boot=tag:efi64,Boot/bootx64.efi
```

### 7. **Slow Network = Timeout Failures** 🐌

```
Reality Check:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
100 Mbps network:  boot.wim (500MB) = 40-50 seconds
1 Gbps network:    boot.wim (500MB) = 4-5 seconds
10 Mbps (old hub): boot.wim (500MB) = 5-6 MINUTES ❌

PXE timeout default: 60 seconds
If transfer isn't complete: TIMEOUT → FAIL
```

**Why This Fails:**
```
Bad cable (Cat5e crimped wrong)
    ↓
Negotiates 100Mbps instead of 1Gbps
    ↓
Large boot.wim takes too long
    ↓
Client PXE ROM: "Timeout waiting for file"
    ↓
BOOT FAILED
```

**Real Example:**
```
User: "Works on PC A, not PC B - same setup!"

Reality:
PC A: Onboard Intel NIC, 1Gbps, good cable → WORKS
PC B: Cheap USB NIC, 100Mbps, bad cable → TIMES OUT

Same server, different client hardware = different results
```

### 8. **SMB Version Incompatibility** 🔌

```
WinPE (Windows 10 1809+):   Uses SMBv3
Old Linux Samba (< 4.11):   Only SMBv1/v2
                                  ↓
           "net use" command FAILS
```

**The Error:**
```cmd
X:\> net use Z: \\192.168.1.1\WinInstall
System error 64 has occurred.
The specified network name is no longer available.
```

**Why:**
```
WinPE tries SMBv3
Samba server: "I don't speak SMBv3"
Connection refused
Share not accessible
Setup.exe can't run
```

**The Fix:**
```bash
# Edit Samba config
sudo nano /etc/samba/smb.conf

[global]
   # Force SMB2/3 support
   server min protocol = SMB2
   server max protocol = SMB3
   
   # Or allow all (less secure):
   server min protocol = NT1
```

### 9. **PXE ROM Not Enabled/Present** 🚫

```
Not ALL computers have PXE capability!

PCs WITHOUT PXE support:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❌ Very old PCs (pre-2000)
❌ Some budget laptops
❌ Tablets/Chromebooks
❌ Macs (without special config)
❌ PCs with disabled network boot option
```

**What You See:**
```
Boot screen:
F12 Boot Menu
┌───────────────────────┐
│ 1. Hard Drive         │
│ 2. CD/DVD Drive       │
│ 3. USB Drive          │
│                       │  ← No "Network Boot" option!
│                       │
└───────────────────────┘
```

**How to Check:**
```
1. Enter BIOS/UEFI during boot
2. Look for Boot Options
3. Check for:
   - "Network Boot"
   - "PXE Boot"
   - "Boot from LAN"
   - "Onboard LAN Boot ROM"

If not present or grayed out:
→ Your PC CAN'T PXE boot ❌
```

### 10. **IP Address Configuration Errors** 🌐

```
CLASSIC MISTAKE:

PXE Server IP:    192.168.1.1/24  ✅
DHCP range:       192.168.0.100-200  ❌ ← WRONG SUBNET!

Result: 
Client gets IP in 192.168.0.x range
Can't reach server at 192.168.1.1
Boot files never download
```

**Another Common Error:**
```
Server has TWO network cards:
- eth0: 192.168.1.1 (connected to clients)
- eth1: 10.0.0.5 (connected to internet)

dnsmasq binds to BOTH interfaces
Responds on wrong interface
Confusion ensues
```

**The Fix:**
```bash
# Force dnsmasq to specific interface
sudo nano /etc/dnsmasq.conf

interface=ens33          # ← Your LAN interface
bind-interfaces          # ← Don't listen on all IFs
```

## 🔥 **Most Common Failure Patterns**

### Pattern #1: "Nothing Happens"
```
Symptoms:
- Client tries to network boot
- Sits at "Searching for boot server..."
- Eventually times out
- No PXE boot

Causes (in order of likelihood):
1. DHCP not running (70%)
2. DHCP running but not sending boot filename (20%)
3. Firewall blocking DHCP (5%)
4. Cable/network issue (3%)
5. PXE not enabled in BIOS (2%)
```

### Pattern #2: "TFTP Timeout"
```
Symptoms:
- Client gets IP address ✅
- Tries to download boot file
- "PXE-E32: TFTP timeout"

Causes:
1. Firewall blocking port 69 (50%)
2. TFTP service not running (30%)
3. Boot file doesn't exist (15%)
4. Wrong TFTP root directory (5%)
```

### Pattern #3: "Boot File Loads, Then Black Screen"
```
Symptoms:
- PXE boot successful ✅
- "Starting Windows..." ✅
- Black screen forever ❌

Causes:
1. BCD configuration wrong (40%)
2. Network driver missing in WinPE (30%)
3. boot.sdi missing/corrupted (20%)
4. RAM too low (< 2GB) (10%)
```

### Pattern #4: "Can't Map Network Drive"
```
Symptoms:
- WinPE loads ✅
- startnet.cmd runs
- "net use" fails with error 53 or 64

Causes:
1. Network driver not in WinPE (50%)
2. SMB share not configured (20%)
3. SMB version mismatch (15%)
4. Firewall blocking port 445 (10%)
5. Wrong server IP in script (5%)
```
